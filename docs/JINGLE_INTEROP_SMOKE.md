# Jingle Interop Smoke

This document records the current interoperability proof for XEP-0166,
XEP-0167, XEP-0176 and XEP-0320.

## Current Automated Smoke

The automated test suite contains an existing-client style Jingle fixture:

- test: `XmppJingleParsesExistingClientInteropFixture`
- file: `tests/Tiedragon.XmppMessenger.Tests/Program.cs`
- command: `dotnet test tests/Tiedragon.XmppMessenger.Tests/Tiedragon.XmppMessenger.Tests.csproj`

The fixture parses the kind of Jingle XML shape produced by mature clients such
as Dino, Gajim, Conversations and Monal:

- `session-initiate` with audio and video contents
- Opus audio payload with RTP parameters
- VP8 video payload plus RTX payload
- `rtcp-mux`
- RTP header extension element preserved as unknown-but-valid XML
- ICE-UDP host, srflx and TCP passive candidates
- DTLS-SRTP fingerprints with `setup="actpass"`
- `transport-info` trickle candidate update
- RTP `session-info` mute event

This proves that the core parser accepts existing-client wire shapes instead of
only parsing XML generated by Teletyptel itself.

## Boundary

This smoke is a code-level interoperability proof, not a claim that the browser
demo can already complete a federated live call with every existing XMPP client.
The current web demo uses the local PHP relay for WebRTC call signaling. A live
external-client proof still needs server-backed XMPP IQ routing, two accounts
and a Jingle-capable client installed.

## Manual Live Smoke Route

Use this when a real external client is available.

Requirements:

- Prosody, ejabberd or another XMPP server with TLS enabled
- two test accounts on the same server
- one Teletyptel test client
- one existing Jingle client, for example Dino, Gajim, Conversations or Monal
- camera and microphone permissions
- STUN/TURN configuration when the clients are not on the same network

Pass criteria:

- both accounts are online and visible through roster/presence
- service discovery reports Jingle RTP, ICE-UDP and DTLS-SRTP support where the
  client exposes those features
- an incoming `session-initiate` is parsed without losing audio/video contents
- `session-accept`, `transport-info`, `session-info` and `session-terminate`
  round-trip through the call state
- mute, camera off/on and hangup are reflected in the call log
- failed NAT traversal is reported as a transport failure, not as malformed XML

Record the client name, version, server, server version and whether TURN was
used when a live smoke is executed.
