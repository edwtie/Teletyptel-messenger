# Accessibility User Testing Notes

Accessibility work must be validated with real users before product claims are
made. The current code provides architecture and testable data flow, not a
completed accessibility product.

## Test Groups

- deaf and hard-of-hearing users using live captions;
- speech-impaired users sending RTT through assisted input;
- keyboard-only users;
- screen-reader users;
- users switching between spoken language, typed text and later sign-language
  experiments.

## Scenarios

- local captions only, no remote sharing;
- captions shared as live RTT;
- final transcript sent as normal chat message;
- remote user reads live text while the sender is still typing;
- user corrects text mid-sentence;
- low-confidence caption is marked clearly;
- speaker label changes during conversation;
- transcript retention disabled.

## Questions To Validate

- Is the delay acceptable for real conversation?
- Are corrections understandable or distracting?
- Is it clear what stays local and what is shared remotely?
- Can users recover from a wrong caption quickly?
- Does the UI work without audio, mouse or visual-only cues?
- Do users trust the agent marker when text was generated by software?

## Privacy Baseline

- Local captions must stay local by default.
- Remote sharing must be explicit.
- Transcript retention must be off by default.
- Agent-generated text must be marked.
- Provider integrations must explain what data leaves the machine.
